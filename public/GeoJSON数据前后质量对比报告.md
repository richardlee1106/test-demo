# GeoJSON 数据前后质量对比报告

## 1. 概述

本报告旨在对比分析“原始 GeoJSON 数据”与“经过处理及新获取的数据”在数据质量、语义丰富度及应用价值上的差异。对比维度涵盖基础属性、语义信息、空间感知能力及 RAG（检索增强生成）适配性。

---

## 2. 数据集定义

- **数据集 A（Before - 原始数据）**：
  - 来源：早期简单的 POI 数据（如 `旅游景点.geojson`）。
  - 特征：字段稀疏，仅包含名称、大类、中类及基础经纬度。
- **数据集 B（After Phase 1 - LLM 增强版）**：
  - 来源：利用 Qwen3-4b 大模型对数据集 A 进行语义推断后的产物。
  - 特征：增加了 `semantic_tags`（适用人群、氛围）、`semantic_description`（功能性描述）。
- **数据集 C（After Phase 2 - 高德详版数据）**：
  - 来源：新爬取的 `wuhan_poi/*.csv` 数据。
  - 特征：包含三级分类、精确地址、商圈信息、评分及价格。

---

## 3. 详细对比分析

### 3.1 字段丰富度对比

| 字段维度     | 原始 GeoJSON (A) | LLM 增强版 (B)              | 高德详版数据 (C)               | 评价                        |
| :----------- | :--------------- | :-------------------------- | :----------------------------- | :-------------------------- |
| **基础信息** | 名称、ID         | 名称、ID                    | 名称、ID、电话、网址           | **C 胜出** (信息更全)       |
| **分类体系** | 2 级 (大类-中类) | 2 级 + LLM 推断的非标准标签 | **3 级** (餐饮-中餐-湘菜)      | **C 胜出** (结构化最强)     |
| **位置精度** | 经纬度、区县     | 经纬度、区县                | 经纬度、区县、**街道、门牌号** | **C 胜出** (适合导航级应用) |
| **空间语义** | 无               | 仅靠文本描述推断 (误差大)   | **商圈 (Business Area)**       | **C 胜出** (关键的空间语义) |
| **评价指标** | 无               | 无                          | **评分 (Rating)、人均 (Cost)** | **C 胜出** (推荐算法核心)   |
| **描述信息** | 无               | **深度自然语言描述**        | 无 (需基于字段生成)            | **B 胜出** (适合直接阅读)   |

### 3.2 语义与空间感知能力对比

#### 🧪 场景测试 1：语义检索 ("适合学生党的平价店")

- **原始数据**：无法检索。没有价格和人群标签。
- **LLM 增强版**：**表现优秀**。模型推断出了 "学生党"、"平价" 标签，可以直接匹配。
- **高德详版**：**表现良好**。虽然没有显式标签，但可以通过 `cost` (价格字段) 进行过滤 (< 20 元)，配合大学周边的 `business_area` 推断。

#### 🧪 场景测试 2：精细空间感知 ("武汉理工南门那家只卖芋泥的店")

- **原始数据**：**失败**。无法区分南门/北门，无法知道卖什么。
- **LLM 增强版**：**失败**。LLM 无法凭空捏造不存在的空间细节和菜单信息（产生幻觉风险）。
- **高德详版**：**潜力最大**。
  - "武汉理工南门" -> 可通过 `address` 字段包含 "南门" 或 `business_area` 匹配。
  - "只卖芋泥" -> **依然困难**，这属于 SKU/评论级数据，即使高德详版也缺失。

---

## 4. 关键发现与结论

### ✅ 进步点 (LLM 增强带来的价值)

1.  **非结构化语义的大幅提升**：将原本冷冰冰的 "餐饮" 标签变成了 "适合聚餐、氛围好、老字号" 等有温度的描述。
2.  **RAG 召回率提高**：对于模糊的意图查询（如"周末去哪玩"），LLM 生成的描述提供了丰富的索引关键词。

### 🚨 痛点 (当前架构的局限)

1.  **空间推理依然薄弱**：LLM 无法解决几何问题。它不知道 "A 在 B 的对面" 除非文本里显式写了。
2.  **数据源天花板**：正如之前分析，**"只卖芋泥" 属于评论/菜单级数据**，无论是原始数据还是高德 POI 均未覆盖。这不是模型问题，是数据源缺失。

### 🚀 建议发展方向

1.  **数据底座迁移**：**全面迁移到数据集 C (高德详版)**。虽然失去了 LLM 生成的描述，但换来了极其宝贵的 `type` (三级分类) 和 `business_area` (商圈)，这两个字段对空间感知的提升远超 LLM 的"猜想"。
2.  **混合增强策略**：
    - 使用 **高德数据** 作为骨架 (位置、分类、评分)。
    - 使用 **LLM** 生成描述 (基于高德的丰富字段生成一段通顺的介绍)。
    - **外挂知识库**：对于特定区域 (如学校周边)，手动维护或爬取更细粒度的 Excle 表 (含"南门"、"特色菜")，作为补充向量库。

---

## 5. 总结

从数据质量看，**CSV 爬取数据 (C) > LLM 增强 GeoJSON (B) >> 原始 GeoJSON (A)**。
建议后续工作流基于 **高德 CSV 数据** 进行清洗和向量化，这将为您的空间智能应用提供最坚实的基础。
